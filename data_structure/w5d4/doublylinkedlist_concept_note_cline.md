# 双向链表（纯概念版）

## 一、为什么需要双向链表

你学过的单向链表，每个节点只有一根指针 `next`，指向后一个节点：

```
[data | next] -> [data | next] -> [data | next] -> NULL
```

这根指针只能带你**一路向前**，想回头？做不到。

单向链表有三个很麻烦的问题：

- **遍历只能一个方向**。从头部走到尾部可以，但从尾部走回头部不行。
- **删除节点需要先知道它的前一个节点**。因为删除的本质是"让前一个节点跳过当前节点，指向后一个节点"，如果你只知道当前节点，不知道前一个节点是谁，链就断了。解决方法只能是遍历一遍找到前驱，O(n) 时间。
- **从尾部操作很困难**。比如你想从尾部插入数据，必须从头遍历到尾，O(n)。

双向链表解决这些问题的方式很简单——**每个节点再加一根指针 `prev`，指向前一个节点**：

```
NULL <-> [prev | data | next] <-> [prev | data | next] <-> [prev | data | next] <-> NULL
```

两边的箭头都能走了。

## 二、结构体结构

```c
typedef struct double_node
{
    datatype data;
    struct double_node *prev;   // 指向前一个节点（前驱）
    struct double_node *next;   // 指向后一个节点（后继）
}dbnode_t, *dbnode_p;
```

和单向链表对比：

| 单向链表节点 | 双向链表节点 |
|-------------|-------------|
| `data` | `data` |
| `next`（1 个指针） | `prev` + `next`（2 个指针） |

每个节点多了一个指针，64 位系统上相当于每个节点多了 8 字节开销。

---

## 三、有头双向链表的结构

采用有头链表（带头节点），头节点不存有效数据，只起定位作用。

### 空链表状态

```
head -> [ 0 | NULL | NULL ]
```

`prev = NULL`、`next = NULL`。

### 有数据时的状态

```
NULL <--- [head] ---> <--- [A] ---> <--- [B] ---> <--- [C] ---> NULL
          prev next          prev next          prev next          prev next
```

几个关键规则：
- 头节点的 `prev` 永远为 `NULL`（没有前驱）
- 第一个有效节点的 `prev` 指向头节点
- 最后一个有效节点的 `next` 永远为 `NULL`（没有后继）
- `NULL` 是两端的终止标记

这个结构可以往两个方向走：
```
正向： head -> next -> A -> next -> B -> next -> C -> next -> NULL（停）
反向： 先走到尾 C -> C->prev -> B -> B->prev -> A -> A->prev -> head（停）
```

## 四、核心操作的概念推演（无代码）

### 4.1 尾插法

**本质：** 找到最后一个节点，在其后面接上新节点。

**指针变化（2 步变 3 步）：**
1. 最后一个节点的 `next` 指向新节点（单向也要做）
2. 新节点的 `prev` 指向最后一个节点（**这步是单向没有的**）
3. 新节点的 `next` 指向 `NULL`（单向也要做）

多做的第 2 步，就是为了以后能从尾部往前走。

### 4.2 头插法

**本质：** 在头节点后面插入，新节点成为第一个有效节点。

**指针变化（2 步变 4 步）：**
1. 新节点的 `next` 指向原来的第一个节点
2. 新节点的 `prev` 指向头节点
3. 如果原来有第一个节点，它的 `prev` 要指向新节点（**这步是单向没有的**）
4. 头节点的 `next` 指向新节点

第 3 步有个**边界情况**：如果链表为空（原来没有第一个节点），这一步跳过不做。因为不存在一个节点来接收 `prev`。

### 4.3 指定位置插入

**本质：** 找到目标位置的前一个节点，在其后面插入新节点。

**指针变化（2 步变 4 步）：**

四步有一个严格的**顺序要求**：必须先让新节点抓住后一个节点（第 1 步），再抓住前一个节点（第 2 步），然后让后一个节点指向新节点（第 3 步），最后让前一个节点指向新节点（第 4 步）。

如果把第 4 步放到第 1 步前面做——即先把前一个节点的 `next` 改成新节点——那么原来的后一个节点地址就丢了，后面的步骤都无法执行。

同样，第 3 步在"插入到最后一个位置时"需要判断：如果后一个节点为 `NULL`，则不需要修改它的 `prev`。

### 4.4 指定位置删除

**本质：** 让删除节点的前一个节点和后一个节点直接连起来，跳过它。

**指针变化（1 步变 2 步）：**
1. 前一个节点的 `next` 指向后一个节点（单向也要做）
2. 如果后一个节点存在，它的 `prev` 指向前一个节点（**这步是单向没有的**）

删除可以细分为三种情况：

**删除第一个有效节点：**
- 头节点的 `next` 指向第二个节点
- 第二个节点的 `prev` 指向头节点

**删除中间节点：**
- 前一个节点的 `next` 指向后一个节点
- 后一个节点的 `prev` 指向前一个节点

**删除最后一个节点：**
- 前一个节点的 `next` 指向 `NULL`
- 不需要修改后一个节点的 `prev`（因为不存在后一个节点）

## 五、双向链表对"删除已知节点"的优化

单向链表如果要删除一个已知的节点指针 `p`，必须先找到它的前驱节点——因为直接做 `p->prev->next = p->next` 在单向链表中不存在，只能遍历。

双向链表直接用 `p->prev` 就拿到了前驱，所以：

```
删除已知节点 p：
  p->prev->next = p->next
  if (p->next != NULL) p->next->prev = p->prev
  free(p)
```

O(1) 完成，不需要遍历。

这是双向链表最显著的优势，也是它被用在一些对删除效率要求高的场景中的原因。

## 六、边界情况汇总

| 操作 | 空链表 | 只有一个有效节点 | 操作第一个节点 | 操作最后一个节点 |
|------|--------|----------------|--------------|----------------|
| 头插 | 不需要调整原第一个节点的 prev（因为没有） | 同左 | 标准 4 步 | 不影响 |
| 尾插 | 新节点成为第一个也是最后一个 | 同左 | 不影响 | 新节点的 next 为 NULL |
| 指定位置插入 | 只能在 pos=0 插入 | pos 只有 0 或 1 | 同头插 | 需要判断后一个节点是否存在 |
| 指定位置删除 | 报错 | 删完后 head->next = NULL | 需要修改第二个节点的 prev | 不需要修改后一个节点的 prev |
| 正向遍历 | 无打印 | 正常打印 | 不影响 | 不影响 |
| 反向遍历 | 无打印 | 正常打印 | 不影响 | 不影响 |

## 七、时间复杂度对比

| 操作 | 单向链表 | 双向链表 |
|------|---------|---------|
| 创建头节点 | O(1) | O(1) |
| 头插 | O(1) | O(1) |
| 尾插（无尾指针） | O(n) | O(n) |
| 指定位置插入 | O(n) | O(n) |
| 删除（已知节点指针） | O(n)（需找前驱）| **O(1)**（prev 直接给出）|
| 删除（按位置） | O(n) | O(n) |
| 正向遍历 | O(n) | O(n) |
| 反向遍历 | 不支持 | O(n) |
| 按值查找 | O(n) | O(n) |

唯一 O(1) vs O(n) 的区别在于"删除已知节点指针"——这是双向链表的灵魂。

---

## 八、双向链表 vs 单向链表总结

| 对比维度 | 单向链表 | 双向链表 |
|---------|---------|---------|
| 节点结构 | `data + next` | `prev + data + next` |
| 指针数 | 1 个 | 2 个 |
| 内存开销（64位） | 数据 + 8 字节 | 数据 + 16 字节 |
| 遍历方向 | 只能从头到尾 | 双向均可 |
| 删除已知节点 | O(n) | O(1) |
| 删除已知节点是否依赖遍历 | 是，需要先找前驱 | 否，prev 直接给出 |
| 插入修改指针数 | 2 个 | 3~4 个 |
| 实现难度 | 简单 | 稍复杂 |
| 主要优势 | 结构简单，内存少 | 删除节点效率高，可双向遍历 |
| 缺点 | 不能反向操作，删除节点麻烦 | 内存开销大，代码复杂一点 |

---

## 九、双向链表适合的场景

**适合用双向链表的情况：**
- 需要频繁删除已知节点的场景（如操作系统中的进程调度队列）
- 需要从尾部往前遍历的场景（如浏览器的历史记录）
- 需要频繁在两端操作的场景

**不适合用双向链表的情况：**
- 内存极度受限的场景（每个节点多一个 8 字节指针，累积起来很可观）
- 只做简单的头插和正向遍历，不需要删除操作

一句话总结：**双向链表是用内存空间换操作效率**——多花一个指针的存储空间，换来 O(1) 删除已知节点和双向遍历的能力。
